## 1. Multipage UI restructure

Today `app.py` renders every section on one long scrolling page. Restructure
into a multipage app. The `dashboard/` modules already separate computation
(figures/data) from rendering, so this is mostly *relocating* the `st.*` calls
out of `app.py` into per-page files — low risk.

### 1.1 Entry point (`app.py`)

Becomes thin: page config, load + filter data once, share via session state,
then define and run navigation.

```python
import streamlit as st
from utils.data_loader import load_data
from dashboard.filters import apply_filters

st.set_page_config(page_title="Logistics Revenue Intelligence", layout="wide")

# Load + filter ONCE; the sidebar filters render for every page and the
# filtered frame is shared with each page via session state.
df = load_data("data/supply_chain_data.csv")
df = apply_filters(df)            # renders sidebar filter widgets
st.session_state["df"] = df

pages = [
    st.Page("pages/overview.py",             title="Overview",            icon="📊"),
    st.Page("pages/trends.py",               title="Trends",              icon="📈"),
    st.Page("pages/spike_investigation.py",  title="Spike Investigation", icon="🔎"),
    st.Page("pages/revenue_drivers.py",      title="Revenue Drivers",     icon="🏆"),
    st.Page("pages/concentration.py",        title="Concentration & Risk",icon="⚠️"),
    st.Page("pages/ask_the_data.py",         title="Ask the Data",        icon="🤖"),
]
st.navigation(pages).run()
```

### 1.2 Pages (`pages/`)

Each page reads `df = st.session_state["df"]` and calls the existing
`dashboard/` helpers:

| File | Content (from current `app.py` sections) |
|------|------------------------------------------|
| `pages/overview.py` | KPI metrics + Revenue by Country + Revenue by Shipment Mode |
| `pages/trends.py` | Monthly revenue trend + freight cost trend |
| `pages/spike_investigation.py` | Spike month selector + country/vendor/shipment charts |
| `pages/revenue_drivers.py` | Top-driver summary, product group, vendor Pareto, country share |
| `pages/concentration.py` | Concentration metrics, dependency score, top-5, scorecard, risk classification, key insights |
| `pages/ask_the_data.py` | RAG Q&A form (+ self-bootstrapping index, section 3.1) |

### 1.3 Notes

- **Shared filters/data:** filters live in the entry script's sidebar and the
  filtered `df` flows to pages via `st.session_state`. Pages never re-load the
  CSV.
- The existing unique `plotly_chart` keys (country share) remain harmless —
  the charts now live on different pages, so they can't collide regardless.
- Risk is low: logic stays in `dashboard/`; pages only render.

## 4. Deployment enablement

### 4.1 Self-bootstrapping vector index (code change)

On a fresh deploy `chroma_store/` does not exist (gitignored), so the
collection is empty and Ask-the-Data returns nothing. Fix: in the
Ask-the-Data page, if the collection is empty, build the index before
answering:

```python
from rag.vector_store import get_collection
if get_collection().count() == 0:
    with st.spinner("Building knowledge index..."):
        from scripts.load_knowledge_base import main as build_index
        build_index()
```

Runs lazily (only on the Ask-the-Data page, on first question), so it does not
slow startup, and makes the app self-provisioning without committing the binary
store. Unchanged locally where the index already exists.

### 4.2 `.streamlit/config.toml`

Basic theme + server config (e.g. headless server settings, theme colors).
Committed so local and deployed runs share configuration.

### 4.3 Deployment docs (in README)

- **Hugging Face Spaces:** the required README YAML frontmatter snippet
  (`sdk: streamlit`, `sdk_version`, `app_file: app.py`, etc.) provided as a
  copy-paste block, plus push/setup steps. HF free tier (16GB) comfortably

- **Streamlit Community Cloud:** connect-repo steps, with an explicit warning

## 5. Screenshots (auto-generated, one per page)

Generate one screenshot per page by driving the running multipage app with a
headless browser, then commit the PNGs to `docs/images/`:

- `overview.png`, `trends.png`, `spike-investigation.png`,
  `revenue-drivers.png`, `concentration.png`, `ask-the-data.png`.
- A one-off capture script (`scripts/capture_screenshots.py`, run manually,
  not part of the app) uses Playwright + the already-cached Chromium to: launch
  the app headless, and for each page click its nav entry, wait for render, and
  screenshot. For Ask-the-Data it types a question, submits, waits for the
  FLAN-T5 answer, then captures. Multipage makes this clean — one capture per
  page, no scroll/section-locating.
- Prerequisite: build the vector index first
  (`python -m scripts.load_knowledge_base`) so Ask-the-Data returns an answer.
- **Playwright is a capture-time tool only** — NOT added to `requirements.txt`
  (keeps the deploy image lean); installed ad-hoc into the venv for the capture.
- The README references `docs/images/<name>.png`; once committed they render.
